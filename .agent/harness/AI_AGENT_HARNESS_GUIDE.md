# AI Agent Harness Execution & Engineering Guide

이 폴더(`.agent/harness`)는 나인티나인 주식회사의 AI 개발 프로젝트를 위한 자동화 검증(하네스) 환경입니다. 하네스 엔지니어링은 단순히 검사 규약을 정의하는 소프트웨어적 절차를 넘어, AI 에이전트가 안전하고 효율적으로 코드를 생성, 실행, 검증할 수 있도록 뒷받침하는 물리적 및 논리적 가상 실험실 환경을 의미합니다.

## 🏗️ 1. 하네스 엔지니어링 4대 핵심 아키텍처

AI 에이전트의 활동 범위를 제한하면서도 필요한 모든 자원을 제공하는 격리된 인프라 체계를 구성합니다.

### 1-1. 가상화 기반의 샌드박스 실행 환경 구축 (Isolation)
에이전트가 코드를 실행하고 테스트할 수 있는 독립된 환경(Docker 등 가상 컨테이너)을 구성합니다. 
* **불변의 인프라**: 에이전트가 생성한 코드가 호스트 시스템의 파일이나 네트워크에 무단 접근하는 것을 방지하며, 실행 후에는 즉시 초기 상태로 자동 복구됩니다.
* **충돌 방지**: 호스트의 글로벌 패키지나 의존성 충돌을 차단하여 신뢰할 수 있는 개발 환경을 제공합니다.

### 1-2. 모델 컨텍스트 프로토콜(MCP) 및 데이터 커넥티비티 (Connectivity)
에이전트가 프로젝트의 실시간 데이터에 접근할 수 있도록 표준화된 정보 통로를 개설합니다.
* **지식 베이스 연동**: 건설 현장의 빅데이터, BIM 설계 도서, 사내 표준 시방서 등을 MCP 서버로 연동합니다.
* **Zero-Shot 진입 지원**: 에이전트가 사전 학습이나 별도의 배경 지식 주입 없이도 최신 데이터를 즉시 불러와 정확한 도메인 지식 기반의 코드를 작성할 수 있게 합니다.

### 1-3. 실시간 입출력 모니터링 및 피드백 루프 시스템 (Self-Healing)
에이전트 조작의 안정성을 담보하기 위해 입출력을 실시간으로 감시하고 교정합니다.
* **I/O 미들웨어**: 명령어 실행 및 표준 출력(stdout), 에러 출력(stderr)을 실시간으로 가로챕니다.
* **자가 수정(Self-Correction)**: 하네스 시스템은 취합된 로그 데이터를 실시간으로 분석하여 에이전트에게 즉각적인 피드백을 전달하며, 에이전트는 이를 바탕으로 오류를 자가 수정하는 반복(Iteration) 과정에 돌입합니다.

### 1-4. 리소스 및 네트워크 권한 제어 (Resource & Policy Control)
에이전트의 오작동을 차단하기 위한 보안 정책 레이어(OPA Policy 등)를 가동합니다.
* **가용성 보장**: AI 에이전트가 사용할 수 있는 CPU 점유율, 메모리 용량, 실행 시간에 임계치(Quota)를 부여하여 인프라 부하를 제어합니다.
* **보안 차단**: 인가된 특정 도메인 외의 외부 서버 접속 통신을 차단하고, 사내 API 호출 시에는 별도의 안전한 인증 토큰 주입을 강제하여 권한 밖의 침해 사고를 원천 차단합니다.

## 🛡️ 2. AI Agent 필수 행동 강령 및 검증 방법

모든 AI 에이전트 및 로컬 개발자는 작업을 마치고 코드를 병합(Merge)하기 전에 **반드시** 이 하네스 룰과 스크립트를 통해 셀프 검증을 수행해야 합니다.

### 검증 스크립트 실행
터미널(Root Directory)에서 다음 명령어를 실행하여 보안 및 프로세스 유효성을 검사하십시오:
```bash
node .agent/harness/validator.mjs
```

### 필수 행동 강령 (Agent Workflow)
1. **작업 완료 전 스캔 강제**: 코드 생성, 수정, 또는 배포 스크립트 작성이 일단락되면 사용자에게 완료를 보고하기 전에 **스스로 `validator.mjs` 명령어를 실행**해야 합니다.
2. **에러 자가 수정(Self-Correction)**:
   - 스캔 결과 `❌ [보안 위반]`이 나타나 민감한 Key나 IP가 소스 파일(`.env` 등 허가된 예외구역 외)에 남아있다면, 즉시 해당 코드를 `process.env.XXX` 혹은 환경변수 참조 방식으로 변환해야 합니다.
   - `⚠️ [권고]` 경고가 나타나 `development_processes_modification_history.md` 업데이트 누락 혹은 `implementation-plan.md` 부재가 발각되었다면, 에이전트는 즉각 프로젝트 변경 내역을 수집하여 히스토리를 갱신하고 PRD를 보완해야 합니다.
3. **Zero-Leakage 무사통과 확인**: 반드시 검증 스크립트가 `🎉 하네스 검증 통과!`를 출력하는 것을 100% 확인한 뒤에만, 사용자에게 작업이 완전히 성공했음을 보고합니다.
4. **코딩 전 사전 확인(Pre-Flight Check)**:
   - 구현 시작 전에 아래 §3의 **"3-1. 코딩 전 사고"** 체크리스트를 수행했는가?
   - 다단계 작업이라면 §3의 **"3-4. 목표 기반 실행"** 계획 템플릿을 사용자에게 먼저 제시했는가?
   - 기존 코드를 수정한다면 §3의 **"3-3. 외과적 변경"** 원칙을 준수하고 있는가?

## 🧠 3. AI 코딩 행동 4원칙 (Coding Behavioral Code)

LLM 코딩 에이전트의 가장 흔한 실수를 체계적으로 방지하기 위한 행동 규범입니다.
이 4원칙이 정상 작동하고 있다면: diff의 불필요한 변경이 줄고, 과도한 설계로 인한 재작성이 감소하며, 질문이 구현 후가 아니라 구현 전에 이루어집니다.

### 3-1. 코딩 전 사고 (Think Before Coding)

> **원칙**: 추측하지 말 것. 혼란을 숨기지 말 것. 트레이드오프를 표면화할 것.

구현을 시작하기 **전에** 반드시 수행:

* **가정을 명시적으로 나열한다.** 불확실하면 질문한다.
* **해석이 여러 가지면 옵션을 제시한다** — 조용히 하나를 선택하지 않는다.
* **더 단순한 접근법이 있으면 말한다.** 필요하다면 역제안(Push-back)한다.
* **불명확한 점이 있으면 코딩을 멈춘다.** 무엇이 혼란스러운지 이름 짓고, 질문한다.

```
❌ 잘못된 예: "사용자 데이터 내보내기 추가" 요청에 조용히 CSV+JSON+파일저장까지 구현
✅ 올바른 예: "내보내기 범위(전체/필터), 형식(JSON/CSV), 전달 방식(API/파일)을 확인하겠습니다"
```

### 3-2. 단순성 우선 (Simplicity First)

> **원칙**: 문제를 해결하는 최소한의 코드. 추측성 기능 금지.

* 요청된 것 **이상**의 기능을 구현하지 않는다.
* 1회성 코드에 추상화(Strategy, Factory, Builder 등)를 적용하지 않는다.
* 요청되지 않은 "유연성"이나 "설정 가능성"을 추가하지 않는다.
* 불가능한 시나리오에 대한 에러 처리를 작성하지 않는다.
* **200줄로 쓴 것이 50줄로 가능하다면, 다시 쓴다.**

```
❌ 잘못된 예: 할인 계산 요청에 DiscountStrategy ABC + PercentageDiscount + FixedDiscount + DiscountConfig 구현
✅ 올바른 예: calculate_discount(amount, percent) 함수 하나로 해결
```

**복잡성은 언제 추가하는가?** → 실제로 필요해질 때. 지금이 아니라 그때.

### 3-3. 외과적 변경 (Surgical Changes)

> **원칙**: 반드시 건드려야 하는 것만 건드린다. 자기가 만든 흔적만 정리한다.

기존 코드를 수정할 때:

* 인접 코드, 주석, 포맷을 **"개선"하지 않는다.**
* 고장나지 않은 것을 **리팩토링하지 않는다.**
* 기존 스타일(따옴표, 타입힌트, 들여쓰기 등)을 **그대로 따른다**, 본인 스타일이 다르더라도.
* 관계없는 dead code를 발견하면 **보고만 하고, 삭제하지 않는다.**

자신의 변경으로 인해 발생한 orphan 처리:

* **자기 변경이 만든** 미사용 import/변수/함수 → 제거한다.
* **기존에 있던** dead code → 요청받지 않는 한 제거하지 않는다.

```
검증 기준: 변경된 모든 줄이 사용자 요청에 직접 추적 가능해야 한다.
```

```
❌ 잘못된 예: 버그 수정하면서 따옴표 스타일 변경 + docstring 추가 + 타입힌트 추가
✅ 올바른 예: 버그를 재현하는 정확한 줄만 수정, 나머지 그대로 유지
```

### 3-4. 목표 기반 실행 (Goal-Driven Execution)

> **원칙**: 성공 기준을 먼저 정의한다. 확인될 때까지 반복한다.

모호한 작업을 **검증 가능한 목표로 변환**한다:

| 모호한 요청 | 검증 가능한 목표 |
|-------------|-----------------|
| "유효성 검증 추가" | "잘못된 입력에 대한 테스트를 작성하고 통과시킨다" |
| "버그 수정" | "버그를 재현하는 테스트 작성 → 수정 → 통과 확인" |
| "리팩토링" | "기존 테스트가 수정 전후 모두 통과하는지 확인" |

다단계 작업 시 반드시 **계획을 먼저 제시**한다:

```
1. [단계 1] → 검증: [확인 방법]
2. [단계 2] → 검증: [확인 방법]
3. [단계 3] → 검증: [확인 방법]
```

**강한 성공 기준**은 에이전트가 독립적으로 반복할 수 있게 한다. **약한 기준**("작동하게 하기")은 끊임없는 확인이 필요하다.

### 3-5. 안티패턴 요약표

| 원칙 | 안티패턴 | 올바른 방법 |
|------|---------|------------|
| 코딩 전 사고 | 파일 형식, 필드, 범위를 조용히 가정 | 가정을 명시적으로 나열, 확인 질문 |
| 단순성 우선 | 단일 할인 계산에 Strategy 패턴 적용 | 실제 복잡성이 필요할 때까지 함수 하나 |
| 외과적 변경 | 버그 수정하면서 따옴표 교체, 타입힌트 추가 | 보고된 문제를 수정하는 줄만 변경 |
| 목표 기반 | "코드를 검토하고 개선하겠습니다" | "버그 X에 대한 테스트 작성 → 통과 → 회귀 없음 확인" |

> **핵심 인사이트**: "과도한" 예시들은 명백히 틀리지 않습니다 — 디자인 패턴과 모범 사례를 따르고 있습니다. 문제는 **시기**입니다: 필요하기 전에 복잡성을 추가하면 코드 이해가 어려워지고, 버그가 늘고, 구현이 느려지고, 테스트가 어려워집니다. **좋은 코드는 오늘의 문제를 단순하게 해결하는 코드이지, 내일의 문제를 미리 해결하는 코드가 아닙니다.**

## 💰 4. 인프라 비용 관리 규칙 (Infrastructure Cost Rules)

> **이 규칙은 2026년 4월 실제 비용 사고에서 도출된 교훈입니다.**
> DB 인스턴스 4개를 인지하지 못한 채 운영하여 월 ₩566,900의 DB 비용이 청구되었으며,
> 이는 전체 인프라 비용(₩655,930)의 **86%**를 차지했습니다.
> 3개 인스턴스를 통합·삭제하여 월 **₩400,000 이상**을 절감했습니다.

### 4-1. 🚨 DB 인스턴스 생성 금지 (Critical Rule)

> **원칙**: 새 DB 인스턴스를 절대 생성하지 않는다. 기존 `nnlogindb` 인스턴스에 테이블을 추가한다.

**NCP Cloud DB for MySQL은 인스턴스 단위로 과금됩니다:**

| 구분 | 월 비용 |
|------|--------|
| DB 인스턴스 1개 | ₩73,000 ~ ₩185,000 |
| DB 인스턴스 4개 (과거) | **₩566,900** |
| DB 인스턴스 1개로 통합 (현재) | **₩160,780** |

**필수 준수 사항:**

1. **새 프로젝트에 DB가 필요한 경우** → `nnlogindb` 인스턴스 내에 테이블을 생성한다.
   ```sql
   -- ✅ 올바른 예: 기존 nnlogindb에 새 테이블 추가
   USE nnlogindb;
   CREATE TABLE new_project_data (...);
   ```

2. **절대 하지 말 것** → NCP 콘솔에서 "DB Server 생성" 버튼을 누르지 않는다.
   ```
   ❌ 잘못된 예: 프로젝트마다 별도 DB 인스턴스 생성 (safetron-mysql, posteval_db 등)
   ✅ 올바른 예: 하나의 nnlogindb 인스턴스에 테이블명으로 프로젝트를 구분
   ```

3. **테이블 네이밍 컨벤션** → 프로젝트별 접두사를 사용하여 충돌 방지:
   - `safetron_nodes`, `safetron_edges` (Safetron AI)
   - `posteval_chunks`, `posteval_documents` (PostEval)
   - `repair_law_embeddings` (Repair Insight Law)

4. **`.env` 설정** → 모든 프로젝트는 동일한 DB 호스트를 사용:
   ```env
   DB_HOST=db-3vog2r-kr.vpc-pub-cdb.ntruss.com
   DB_NAME=nnlogindb
   DB_USER=ninetynine99
   ```

### 4-2. 인프라 비용 점검 체크리스트

새로운 클라우드 리소스를 생성하기 전에 반드시 확인:

| 점검 항목 | 확인 |
|-----------|------|
| 기존 리소스로 해결 가능한가? | □ |
| 이 리소스의 월 과금 단위와 예상 비용은? | □ |
| 사용 종료 후 삭제 계획이 있는가? | □ |
| 관련 ACG/보안그룹도 함께 관리되는가? | □ |

### 4-3. 현재 인프라 비용 구조 (2026-04-28 기준)

| 서비스 | 월 비용 | 비고 |
|--------|--------|------|
| Server (VPC) - ninetynine hub | ₩79,190 | VM 서버 |
| Cloud DB for MySQL - nnlogindb | ₩160,780 | **유일한 DB (통합 완료)** |
| Block Storage | ₩6,220 | 디스크 |
| Public IP | ₩3,620 | 고정 IP |
| **월 합계** | **~₩250,000** | 통합 전 ₩655,930 대비 **62% 절감** |
